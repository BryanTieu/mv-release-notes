## **Rocket UniVerse Hotfix** 

## **Version 11.2.3.4648** 

## **Applicable Platforms** 

UniVerse 32-bit on Windows only 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at U2Support@rocketsoftware.com. 

## **Description of the problem** 

|**Issue number**|**Description**|
|---|---|
|UNV-22033|Prior to this release, the use of Windows UNC names to identify shared<br>files may have caused UniVerse to abnormally terminate. The problem<br>was due to a memory allocation error which occurred when the UNC file<br>was opened. The memory allocation error may have caused UniVerse to<br>terminate at some point after the file open. Repeated instances<br>(hundreds or thousands) of the failure could eventually impact unrelated<br>UniVerse sessions and require UniVerse and/or Windows to be restarted.<br>Thisproblem has been resolved at the current release.|



## **Installation instructions** 

Install as a typical installation. Refer to the _UniVerse Installation_ manual for details.

## **Version 11.2.3.4646** 

## **Applicable Platforms** 

UniVerse 64-bit on Linux only 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at U2Support@rocketsoftware.com. 

## **Description of the problem** 

|**Issue number**|**Description**|
|---|---|
|UNV-18521|Prior to this release, under certain conditions PHANTOM processes might<br>remain in the defunct state until the parent process exited. If the parent<br>process encountered an error while spawning a phantom, then all<br>phantoms started by the parent after the error would remain in the<br>defunct state when they completed. The NOTIFY setting would not alter<br>the behavior and the parent process would need to exit in order to have<br>the defunct processes exit. This problem has been resolved at the<br>current release.|
|UNV-18532|Prior to this release, the use of the MQDISC BASIC function could result<br>in defunct processes being left behind. The defunct processes were<br>initiated as u2mqi processes that were created when the connection is<br>opened. The defunct process could remain in the process table until the<br>original uvsh session exited. At the current release, the processes will<br>now exit completelywhen the MQDISC function is used.|



|**Issue number**|**Description**|
|---|---|
|UNV-19046|At UniVerse 11.2.3, a change was made to the UniVerse startup script on<br>UNIX/Linux where the UniVerse Transaction Logging daemon was only<br>started if TXMODE was enabled. This could have resulted in WRITE<br>failure messages similar to the following:<br>Write of 9999999 will not occur.<br>Contact System Administrator.<br>The failure could occur when TXMODE was set to 0 and UniVerse files on<br>the system had previously been activated for Transaction Logging. The<br>problem has been fixed at this release by initializing the transaction<br>logging daemon during UniVerse startup regardless of the TXMODE<br>setting.<br>Note, another problem with the original change was reported under<br>UNV-18918 and has alreadybeen corrected in the 11.2.3.4640 release.|
|UNV-19226|Prior to this release, some users found that performing updates to the<br>UniVerse global catalog space caused processes to hang if another user<br>already held an operating system 'flock'  on the catalog space. This issue<br>was unique to the Linuxplatform and has been resolved at this release.|
|UNV-19347|At UV 11.2.3, the uv.rc script sent output incorrectly to a /dev.null file<br>rather than the/dev/null device. This issue is now fixed.|
|UNV-19405|Prior to this release, U2 Replication failed to suspend and sync on Linux<br>11.2.3 64-bit operating systems. Some replication groups would fail with<br>"publistener process aborted" error messages during the sync. This issue<br>has been resolved.|
|UNV-19454|An issue was encountered on the initial 64-bit port of UniVerse 11.2.3 on<br>Linux related to the execl() system call. The problem was related to a<br>difference in behavior between 32-bit and 64-bit Linux. The specific<br>problem identified was associated with U2 Replication but all instances<br>of the execl()system call have been resolved at this release.|
|UNV-19471|In rare cases, MQI processes may have hung on a<br>futex/FUTEX_WAIT_PRIVATE call. This problem was caused by improper<br>signal handlingand is now fixed.|



## **Installation instructions** 

Install as a typical installation. Refer to the _UniVerse Installation_ manual for details.

## **Version 11.2.3.4641** 

## **Applicable Platforms** 

UniVerse 64-bit on Linux only 

## **Description of the problem** 

|**Issue number**|**Description**|
|---|---|
|UNV-18865|Prior to this release, the following two parameters were incorrectly<br>defined in the U2MQI.H INCLUDE file:<br>EQUATE MQPMO_NO_SYNCPOINT TO 64 ;* 0x40<br>EQUATE MQGMO_NO_SYNCPOINT TO 64 ;* 0x40<br>These are both MQLONG parameters, which are 4-byte integers, not 64-<br>bytes. Both parameters have been updated to reflect the correct values,<br>and are now defined as"<br>EQUATE MQPMO_NO_SYNCPOINT TO 4 ;* 0x4<br>EQUATE MQGMO_NO_SYNCPOINT TO 4 ;* 0x4<br>This issue has been resolved.|
|UNV-18907|Prior to this release, when MQPMO and MQGMO structures were used<br>and an MQGET call was issued, the MsgId contained eight extraneous<br>characters at the end of the string. This issue has been resolved.|
|UNV-18908|Prior to this release, when MQPMO and MQGMO structures were used<br>and an MQGET call was issued, the CORRELID would have an extraneous<br>CHAR(255) at the end of the string. Further, if this value was encoded<br>with Base64A there were two extraneous characters added at the end of<br>the string. This has been resolved.|
