## Rocket UniVerse Hotfix 11.2.4.4714 

## Version 11.2.4.4714 

## Applicable platforms 

- 64–bit AIX 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at U2Support@rocketsoftware.com. 

## Description of the problem 

|**Issue number**|**Description**|
|---|---|
|UNV-21627|Prior to this release, under certain conditions it was possible for a BASIC<br>program using the MQ interface to hang. If the program was executing an<br>external command such as LIST.READU while an MQ connection was open, a<br>signal handling issue could cause the BASIC program to hang. When the child<br>LIST.READU process sent a signal as notification of completion, the active u2mqi<br>process might incorrectly receive the signal intended for the parent uvsh process.<br>When this happened, the parent uvsh process would hang waiting for that signal.<br>Due to the nature of the problem, encountering this behavior could be very<br>random. UniVerse has been modified to ensure that signals intended for the<br>parent uvsh process are now correctly received. This issue is resolved.|

## Description of the problem 

|**Issue number**|**Description**|

|UNV-20870|Prior to this release, when using the terminfo "iprog" capability to run a terminal<br>initialization program, UniVerse would incorrectly display the following message<br>when setting the terminal type. "waitpid(): No child processes" at line 172 of<br>set_term.c. This issue has been resolved.|

This behavior could have resulted in<br>subsequent child processes hanging if the pid of the new process matched a pid<br>that was incorrectly left in the job queue. This issue has been resolved.|

The phantom process would start but would immediately<br>hang without performing any processing. When the parent UniVerse session<br>terminated, the child phantom processes would remain in the process table with<br>a parent process id of 1. The problem was only reproducible on the HP Itanium<br>platform and has been resolved.|

|UNV-21269|At 11.2.x on the HP Itanium platform, some customers experienced situations<br>where RD or WR group locks were not being properly released. User processes<br>requiring those locks would hang until the group locks were made available by<br>a manual release. The behavior was related to a synchronization problem which<br>could occur in a multi-CPU environment. The behavior was experienced in large<br>user environments when the system was under heavy load. This issue has been<br>resolved.|

## Version 11.2.4.4711

## Description of the problem 

|**Issue number**|**Description**|
|---|---|
|UNV-20548|Prior to this release, when SQL empty-null mapping was turned on, empty<br>strings in text-based fields were converted into a single space in the UniVerse<br>SQL terminal or printer output. This made it impossible to distinguish between<br>records that were null and those with a valid single space as the actual data.<br>Beginning at this release, a new option, NULL, has been added to the SET.SQL<br>command.<br>▪<br>When the SET.SQL NULL option is followed by an empty set of quotes, the<br>value is set to the default space value. For example,**SET.SQL NULL** _""_sets<br>the value to a single space.<br>▪<br>When the SET.SQL NULL option is followed by a character enclosed in<br>quotation marks, the value is set to the specified value. For example,**SET.SQL**<br>**NULL** _"some character"_sets the value to_some character_.<br>▪<br>When the SET.SQL command is not followed by an option, it displays the<br>current setting for all options.|

Rocket UniVerse Hotfix 11.2.4.4710 

|**Issue number**|**Description**|
|---|---|
|UNV-19579|An enhancement has been made to the Audit log. Beginning at this release, it is<br>possible to include the full path of the program being executed in the Audit log<br>file. You can also optionally include the the stack of programs to the Audit log file.<br>There are two global configuration policies:<br>`log_program_path= ON | OFF default to ON`<br>`log_program_stack=ON | OFF default to ON`<br>These two policies are independent of one another.<br>If log_program_path is ON, all program/subroutine names will be full path.<br>For example,<br>`Program........... /eskimo3/uv112/catdir/*GSUB2.19579/FSUB2`<br>If log_program_stack is ON, the full execution stack will be logged with the<br>current program path. The PROGRAM field will be a multi-value field. The first<br>value is the current program, the second value is "Call Stack:", the rest of values<br>are stack contents in reverse calling order, starting from the most recent called<br>program/subroutines, to the main program. For example,<br>`Program.................. /eskimo3/uv112/catdir/*IS.FULLPATH`<br>`. Call Stack:`<br>`. /eskimo3/uv112/catdir/*GSUB2.19579/FSUB2`<br>`. /eskimo3/uv112/catdir/*GSUB2.19579/FSUB1`<br>`. /eskimo3/uv112/catdir/*GSUB2.19579/`<br>`. /home/jcui/uvtest/BP.O/19579/SUB2`<br>`. /home/jcui/uvtest/BP.O/19579/SUB1`<br>`. /home/jcui/uvtest/BP.O/19579/`<br>NOTE: If you have nested or recursive programs, this log can grow very long and<br>overall performance may be negatively impacted.|
|UNV-19580|Beginning at this release, when using Audit logging to log database updates,<br>the audit logs have been enhanced to distinguish between a new record<br>being added (insert) to the database and an existing record being modified<br>(update). For example, the audit log entry will now contain either WRITE:ADD or<br>WRITE:UPDATE to distinguish the type of write operation performed.|
|UNV-19630 and<br>UNV-19631|An enhancement has been made to the DAT.QUERY.COMMAND and<br>DAT.QUERY.RESULTSET events. Now, when the DAT.QUERY.RESULTSET event<br>is enabled, the DAT.QUERY.COMMAND audit log record contains the command<br>executed and the result set containing all record IDs that were returned by the<br>command. Note, the individual DAT.QUERY.RESULTSET audit log entries that<br>contained the actual data returned by the command are no longer written to the<br>audit log.|
|UNV-20288|For Linux and HP Itanium only at UniVerse 11.2.4, if the LANG environment<br>variable was set to a value other than C, the unirpcservices file might be modified<br>incorrectly on an 11.2.4 upgrade. Starting at this release, the LANG setting is<br>temporarily changed to C during the installation process, which allows the<br>unirpcservices file to be updated properly, fixing this issue. The workaround at<br>UniVerse 11.2.4 is to manually set LANG=C prior to running uv.load.|
