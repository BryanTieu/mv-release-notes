Problems Fixed in UniVerse Release 10.3.12 (Build 4243)

Issue Short Description ----- -----------------

UNV-4168 UniVerse -- By default, after installing Universe on Windows
platforms, nonadministrator users could not run any globally cataloged
programs. This problem occurred due to insufficient permissions on the
catdir directory, which is used to store UniVerse cataloged programs.
The installation process has been modified to ensure the catdir folder
has update permissions for all users, resolving the problem.

UNV-4350 UniVerse -- When a member of the Administrators group logged on
to UniVerse, the previous account name or path name used on the previous
login is expected to be remembered. On User Account Control (UAC)
systems, the previous information was incorrectly not being remembered
on login attempts. This problem has been resolved.

UNV-4708 UniVerse -- Prior to this release, when upgrading from UniVerse
10.2 to UniVerse 10.3, the installation process might generate Automatic
Data Encryption (ADE) errors related to migrating ADE information to the
new release. This could happen even if ADE had never been used at the
UniVerse 10.2 release. Changes have been made to the installation script
at this release to detect whether ADE had been used at UniVerse 10.2.

                If the installation process determines that ADE had been
                used, UniVerse tries to transfer the ADE information
                and will display messages about the success or failure
                of the process. If the installation process detects
                that ADE had not been used at UniVerse 10.2, a message
                is displayed indicating that no ADE transfer was
                performed. The &KEYSTORE& and &ENCINFO& files from
                UniVerse 10.2 are copied to &KEYSTORE&102.backup and
                &ENCINFO&102.backup, respectively. Therefore, when the
                installation completes, no ADE is set up for UniVerse 10.3
                if the ADE functionality had never been used at
                UniVerse 10.2.

UNV-5136 XAdmin -- Before this release, on UAC-enabled systems, after
configuring telnet through XAdmin, the telnet settings were not saved
when logged in as an admin user, and the following error message
appeared:

            "The subroutine failed to complete successfully."

        In order to the save the changes, the user had to be
        logged in as an Administrator. This problem has been fixed.

        In addition, UniVerse previously stored the port number in
        the \windows\system32\drivers\etc\services file. Permission
        problems may have been encountered when attempting to 
        update this file. To solve this problem, the port number
        is now written to the Registry.

UNV-5218 On Windows platforms, a new version of the uvdiag utility is
available at this release.

string comparisons. Under certain conditions, a string comparison
operation could result in an unhandled exception violation on Windows
platforms or a core dump on UNIX platforms. An example condition is when
strings involved in the comparison were of a format similar to 430667E4
or 4308756E-2. The 'E' followed by a numeric would result in UniVerse
treating the string as a number using scientific notation. A memory

inconsistent. For example, an operation which did fail would likely run
correctly the next time it was executed. The memory error has been fixed
at the current release.

UNV-4358 UniVerse -- When resizing a file with the CONCURRENT option on
UNIX platforms, the UniVerse BASIC WRITE statement may have occasionally
taken the ELSE clause.

                This problem was caused by a potential race conditon
                which occurred when the temporary resize file was being
                moved back to the original file name. Additionally,
                UniVerse may have displayed inaccurate messages when
                attempting to open a file during such a race condition.
                At the current release, error messages will be
                suppressed when the first attempt to open a file fails
                and a systemwide resize lock has been obtained. Retry
                logic has been implemented to perform the open a second
                time if the first attempt fails and a systemwide resize
                lock has been obtained. If the second attempt fails, the
                error messages will no longer be suppressed.

                In addition, a resize.log file is now created in the
                $UVHOME directory. This log file contains messages
                regarding the stages of the resize operation, including
                start, temporary file creation, and complete. The
                following example illustrates the contents of the
                resize.log file:

                Fri May 27 16:56:15 2011 blao(1044666) in account
                "/home/blao/tuv": Resize Started (file path: "TEST",
                inum: 61181) with temp file "resizekLYlEa".
                From 3 17 2
                To 30 13 4 CONCURRENT

                Fri May 27 16:56:16 2011 blao(1044666) in account
                "/home/blao/tuv": Temp file Created (file path:
                "resizekLYlEa", inum: 69696).

                Fri May 27 16:56:17 2011 blao(1044666) in account
                "/home/blao/tuv": Resize Completed.

                The following error messages have been added for a
                resize operation:

                RESIZE: FL_GRPRESIZ bit was set on group 0. Not resizing file.
                RESIZE: Error on TEST. File not resized.

                Changes have been made to RESIZE to ensure that an
                existing RESIZE operation has fully completed prior to
                allowing another RESIZE operation to start. At prior
                versions, there was a small timing window where a second
                version could start while the first version was
                finishing. If this situation is detected, the above
                error messages will be displayed and second RESIZE
                operation will not continue.

UNV-4461 UniVerse BASIC -- If a WRITE is done to an invalid file pointer
within a UniVerse BASIC subroutine, the error is fatal and will cause
the program to abort. If such a fatal error occurs within a transaction,
the transaction would be rolled back and transaction levels updated when
the fatal error caused the process to return to TCL.

                If, however, the subroutine was called through the
                UniObjects server, such a fatal error would cause
                control to return to the client program, bypassing
                TCL where the cleanup is normally done. This behavior
                of returning to the client program on a fatal error is normal.
                However, in this situation the result was that the
                transaction cleanup was not done, and transaction levels
                were left in an inconsistent state.

                The UniObjects server has been modified so that when
                a fatal error occurs, it will now go through the
                appropriate transaction cleanup if a transaction was
                active.

11028 Prior to this release, the RESIZE command ignored a GROUP.SIZE

11058 Before this release, the UniVerse DEACTIVATE.ENCRYPTION.KEY
command and the UniVerse BASIC DEACTIVATEKEY statement returned a
successful message regardless of the current activation status of the
key. At this release, if the key is already deactivated, the

UniVerse BASIC DEACTIVATEKEY statement will set the STATUS() variable

11161 Prior to this release, installing UniVerse on Windows platforms
would fail if the Administrator was a member of too many groups. The
installation would produce a misleading message error message indicating
"You must have Adminstrative privileges on this system to install
UniVerse." The error was a memory related issue and UniVerse has been
modified to dynamically handle the number of member groups.

11220 Prior to this release, UniAdmin displayed the error "Unable to get

release, UniVerse has been modified to place the services file in the
appropriate Windows system directory which resolves this issue.

    Additionally, the UniVerse Control Panel has been converted

11325, Prior to this release, the BUILD.INDEX command failed on 11454
Pick-style 'A' or 'S' type dictionary index items that had been created

11362 Beginning at UniVerse 10.2.12, if a UniVerse file contained one or
more free buffers on the free buffer chain, using the FORMAT.CONV or
fnuxi command to convert the machine class of the file failed and
potentially damaged the file. This problem was caused by an internal
error related to 32-bit vs. 64-bit conversion, and has been fixed.

11496 When executing the FORMAT.CONV or fnuxi command directly against
the DATA.30 portion of a dynamic file, UniVerse processed the file as a
static hashed file, leaving the file corrupted. The FORMAT.CONV and
fnuxi commands will now restrict processing to the complete dynamic
files. Attempting to process individual elements (such as DATA.30), will
result in a message indicating that the operation is not allowed.

11648 At UniVerse 10.3, locks on records and files will affect the
XMLTODB function in the following ways:

    When using the XMLTODB function, if a user holds a
    READU lock on an individual record and the XMLTODB
    function tries to issue a READU lock on the same record,
    that READU will wait until the record is released.

    If a user holds a READU lock on any record in the file and
    the XMLTODB function tries to set a FILELOCK on the file,
    the FILELOCK failed and issued a message similar to the
    following example:

        Error when setting the FILELOCK on <filename>,
        conflict with user "#" XMLTODB failed.
