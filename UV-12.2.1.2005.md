## General Hotfix information 

This release notes document contains a cumulative summary of all the Hotfixes made available for a specific Generally Available (GA) product version. Please browse this document to find the Hotfix release that fits your specific needs. 

Hotfixes are identified as "Controlled Releases" within Rocket Business Connect (RBC). Hotfixes are only visible by searching for the specific version and build number in the Controlled Release field. 

To obtain a specific Hotfix: 

1. Log into the RBC system and modify your existing license. 

2. Click **Edit** next to your existing product, and then click **Search** to view the list of GA product releases. 

3. After the product availability matrix displays, enter the Hotfix version number (which includes the product version and build number, for example 11.3.1.6005) in the **Controlled Release** field just above the table of product releases. 

4. Click **Search** . 

The Hotfix release will display on the product matrix, and can be selected for ordering and download. If you have questions about RBC, please contact us at U2Support@rocketsoftware.com. 

**Note:** Hotfix releases are intended to be short-term solutions and have had limited testing in order to be made available quickly. All Hotfix changes will be included in the next GA release, and will be fully tested by Rocket’s Quality Assurance (QA) team across all supported platforms and environments. It is strongly recommended that after taking a Hotfix, customers upgrade to the next available GA release to ensure the highest quality experience. 

For questions, please contact support at U2Support@rocketsoftware.com.

## Rocket UniVerse Hotfix 12.2.1.2005 

February 2025 

## Applicable platforms 

## ▪ Windows 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at support@rocketsoftware.com. 

## Description of the problem 

|**Issue**<br>**number**|**Description**|
|---|---|
|UNV-34269|Prior to this release, upgrading an RFS enabled UniVerse system may have failed under<br>certain conditions. The conditions involved changing the N_BIMG or N_AIMG uvconfig<br>settings from the default value of 2. This resulted in the failure of some processes to<br>start during the install which caused the upgrade install to fail. This issue has been<br>resolved in the current release.|
|UNV-34345|Prior to this release, dynamic files may have been corrupted during RFS recovery. This<br>issue has been resolved in the current release.|
|UNV-34453|Windows only. Prior to this release, the RFS recovery process may have failed with<br>`undo failed`messages being written to the`uvsm.log`. When this occurred,<br>UniVerse was unable to restart until a`uvcntl_install`command was executed to<br>clear the RFS logs. This issue has been resolved in the current release.|
|UNV-34459|Prior to this release, RFS recovery may have failed when processing a`CLEAR.FILE`<br>operation. This issue has been resolved in the current release.|
|UNV-34460|Windows only. Prior to this release, RFS recovery could fail when recovering a file which<br>was larger than 4GB. The problem was related to addresses > 4GB not being handled<br>correctly and has been resolved in the current release.|
|UNV-35575|Prior to this release, an`invalid type 99`message could be generated in the<br>`uvsm.log`file during RFS recovery. It was determined that this message was<br>unnecessary and will no longer be displayed in the current release.|
|UNV-35683|Windows only. Prior to this release, the`DBopen`routine executed when opening a file<br>leaked a small amount of memory. This behavior could cause a problem during RFS<br>recovery, depending on the number of files processed. If a large number of files are<br>being opened during startup, available memory could be exceeded, causing the startup<br>process to terminate unexpectedly. This issue has been resolved in the current release.|
|UNV-35695|Windows only. The startup process on Windows includes a timer to detect if the startup<br>process is taking too long and should be terminated. With the introduction of RFS<br>recovery, the timer could be exceeded if the recovery involved processing a large<br>number of files. In this situation, the startup process would incorrectly terminate<br>before completing all recovery operations. In the current release, the startup process<br>recognizes that an RFS recovery is in process and will not incorrectly terminate the<br>startup during the recovery operation, thus resolving this issue.|
|UNV-35734|Windows only. Prior to this release, the RFS recovery process could take a long time to<br>complete. The performance issue was related to the opening and closing UV files for<br>each redo operation. In the current release, the files being processed will remain open<br>during the recovery process to avoid the performance impact of opening, closing, and<br>reopening file to perform the redo operation.|

|---|---|
|UNV-33616|Prior to this release, adding an ALL.NULLS index to a file which already includes other<br>virtual field indexes could result in the virtual field indexes being updated incorrectly.<br>Virtual field indexes include "I" type and "C" type indexes. Including the`FORCE`<br>option on the`CREATE.INDEX`command line will allow the process to execute<br>while other users have the file open. The default behavior of`CREATE.INDEX`when<br>SYSTEM_BUFFER is ON is for the command to fail to execute if the user does not have<br>exclusive access to the file. The`FORCE`option can also be used with`DELETE.INDEX`<br>to override the requirement of no other users having the file open.|

**Important:** Vulnerabilities were discovered in the UniRPC services which could be exploited to: 

- perform an ROP (Return-Oriented Programming) attack

The above 3 vulnerabilities have been resolved in this release.

## Rocket UniVerse Hotfix 12.2.1.2001 

## January 2023 

## Applicable platforms 

- Windows 

- AIX 

- Linux 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at support@rocketsoftware.com. 

## Description of the problem 

|**Issue**<br>**number**|**Description**|
|---|---|
|UNV-33252|Prior to this release, upgrading to 12.2.1 from an 11.3.5 version would fail due to the<br>uvconfig parameters_SP_USPLOG_GROUP_and_SP_USPLOG_MERGE_INTERVAL_. These pa-<br>rameters were included in 11.3.5 as part of a spooler performance enhancement which<br>has not yet been implemented at 12.2.x. In this release, these parameters have been<br>added to uvconfig to avoid the upgrade issue. The uvconfig parameters are place hold-<br>ers and do not currently perform any function. The spooler performance enhancement<br>will be included at a later 12.2.x version.|
|UNV-33306|Beginning with 12.1.1, the`FIX`option of the`VERIFY.SQL SCHEMA`command did<br>not behave correctly. The command would generate an error and incorrectly delete the<br>schema. This issue has been resolved.|
|UNV-33390|Beginning with 12.1.1, a memory related issue could cause a process to terminate while<br>in a critical level of code and force a UniVerse shutdown. The problem could happen<br>when the BASIC call stack of a uvsh process was many layers deep. In some instances,<br>unassigned memory may have been accessed when returning to a prior level and this<br>could cause the process to terminate. This issue has been resolved.|
