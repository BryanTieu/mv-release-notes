## **About this release** 

- UniVerse 14.2.1 will be installed with RFS Mode Off by default. Users can enable RFS by modifying the configuration. Only files protected by RFS will be cached to the System Buffer. For more information, see the "RFS Mode" chapter. 

- As previously announced in the 14.1.1 release, RFS two process mode has been removed, and this remains the case in 14.2.1. 

- Also as introduced in 14.1.1, on AIX and Linux systems the existence of the uvdb user is required for installation and the UniVerse daemon processes will continue to run as this user in 14.2.1. 

- The Python release packaged with UniVerse 14.2.1 has been upgraded to 3.13 with this release. 

   - **OpenSSL module requirements:** 

With the Python 3.13 release, the Python SSL Module supplied with 14.2.1 is based on OpenSSL 3.0.x. To use this module, OpenSSL 3.0.x must be installed on your Windows, Linux, or AIX operating system. The only exception is Red Hat Enterprise Linux (RHEL) 8 where the SSL module is compatible with OpenSSL 1.1.1k. OpenSSL 1.1.1k is the default version of OpenSSL installed with the RHEL 8 distribution. 

If the installed version of OpenSSL does not match the Python SSL Module version of 3.0.x, compatibility errors will be encountered and attempts to use the functionality will fail. 

   - **Executing u2py.run from a Windows 11 console shell:** Windows 11 introduces Windows Terminal as the default terminal for the shell environment. The Python 3.13 release introduces the new interactive shell PyREPL. The combination of these two will result in a compatibility issue when using `u2py.run` . This can be resolved through the use of a Windows environment variable. Setting the Windows environment variable PYTHON_BASIC_REPL to any value resolves the compatibility issue between PyREPL and the Windows Terminal environment. Deleting the environment variable will revert back to the PyREPL interactive shell. 

- Group Access Concurrency Control. In UniVerse 14.2.1, the `GSEMNUM` and `GLTABSZ uvconfig` parameters have been reintroduced to restore group access concurrency control functionality from Release 11.x. This ensures that unprotected file I/O in 14.2.1 behaves consistently with the I/O characteristics of unprotected files in 11.x, maintaining compatibility with earlier releases. This functionality applies to all files when RFS_MODE is set to 0, and to files not marked as protected when RFS_MODE is set to 1.

## **RFS mode** 

In 14.2.1, the Recoverable File System (RFS) mode is available. 

By default, RFS mode is set to OFF. Users can enable RFS as needed by setting the RFS_MODE parameter in the `uvconfig` file to 1. When enabled, files can operate in two distinct modes: Protected Files and Unprotected Files. The following sections outline the differences between these modes and the configuration options available for RFS. 

## **Unprotected Files (RFS OFF)** 

- All UniVerse files are unprotected when RFS mode is OFF. 

- When a UniVerse file is opened, an AFT (Active File Table) entry is created with key information such as inode and dno numbers. This information is used for concurrency control; file header information is not stored in the AFT entry. 

- File activity is not cached to the System Buffer. File I/O operations are directly applied to the physical file, consistent with behavior in UniVerse releases prior to v12.1. 

- An AFT entry is removed from the table if it is not opened by any process. 

## **Protected Files (RFS ON)** 

- By default, files are unprotected even when RFS mode is ON. Files must be explicitly configured as protected. 

- When a protected file is opened, an AFT entry is created with most of the file header information, which is stored in the AFT entry. 

- When a file is configured as protected, the file activity is cached to the System Buffer, and File I/O operations are applied to the System Buffer instead of directly to the physical file.

## **Additional Notes** 

Additional Notes about configuring protected files using the `rfs_files` configuration file: 

- Files being protected must physically reside in the UniVerse account specified by the ACCOUNT phrase. Files defined by a remote 'F' pointer or 'Q' pointer will be ignored. This is true when using LEVEL=ACCOUNT to define an account with account level protection (and when individually defining protected files with a FILE= phrase). This behavior exists to avoid conflicts where a file might be defined differently in the remote account compared to the actual account. 

- If a file is defined with multiple FILE= phrases, only the first FILE= phrase will take effect. This would be true if the same VOC file name were used or if a different VOC name pointing to the same file was used. 

- The FILE= phrase is only valid when the account is configured for FILE level protection. The EXCLUDED_FILE phrase is only valid when the account is configured for full ACCOUNT level protection. 

- When the EXCLUDED_FILE phrase is used, the excluded file will be based on the VOC file name. When the VOC file is being processed, the file name specified in the EXCLUDED_FILE phrase will be ignored. 

**Note:** If multiple VOC pointers define the same physical file, all pointers will need to be excluded for the file to remain unprotected. 

- When a new file is created using `CREATE.FILE` in an account set up for ACCOUNT level protection, the file will automatically be marked as protected. However, if the new file name matches any of the exclusion rules defined in the `rfs_files` configuration file for that account, the file will be considered unprotected. 

- When a new file is created using `CREATE.FILE` in an account set up for FILE level protection, the file will be marked as protected if a FILE= phrase matching the filename exists in `rfs_files` . Otherwise, the new file will be unprotected. 

- Files must exist in a valid UniVerse account in order to be marked as protected. That is, files residing in nonUniVerse folders or directories will need to reside in a UniVerse account in order to be configured as protected.

## **System buffer runtime recovery framework** 

Starting with UniVerse 14.1.1, the system buffer runtime recovery framework was introduced to ensure consistency and integrity in critical regions of the database, improving the stability of the UniVerse server. ## **Index awareness** 

UniVerse 14.2.1 introduces "Index Awareness," a significant enhancement to how UniVerse handles index updates. 

## **What is this new feature?** 

The Index Awareness Implementation introduces improvements that will now allow index file changes to be recognized by users who already have the data file open. These changes include for example `CREATE.INDEX` , `DELETE.INDEX` , `ENABLE.INDEX` , and `DISABLE.INDEX` . 

## **How does this differ to the previous functionality?** 

Currently, when a UniVerse file is opened by a UniVerse process, all the information about its associated index file is loaded at that point in time. This includes information on all existing indexes and whether or not the index is enabled or disabled. If changes are made to the index while the file is open, the changes are not recognized until the file is closed and reopened. 

## **What improvements will I see?** 

This new feature ensures indexes remain consistent when changes are made to an index while existing UniVerse processes already have the data file open. Users are no longer required to close and reopen the data file for the changes to be recognized. 

## **Concurrent Index Operations:** 

Before UniVerse 14.2.1, index-related commands such as `CREATE.INDEX` , `BUILD.INDEX` , and `DELETE.INDEX` would fail if the data file was open in another process. Starting with UniVerse 14.2.1, this restriction has been removed, regardless of whether the system operates in RFS Mode or Non-RFS Mode. 

## **New TCL command VERIFY.INDEX:** 

This command verifies data and index file synchronization. While `VERIFY.INDEX` does not require exclusive file access, it relies on a consistent snapshot of the data. Therefore, it must be run offline (not concurrently) to prevent inaccuracies due to ongoing changes.

## **Fixtool improvements** 

Beginning with 12.1.1, the `fixtool` command required exclusive access to a file in order to execute. This was true even when only scanning the file to check for errors. Additionally, when used with RFS protected files, only the physical file was scanned. This may have led to inconsistencies if the system buffer contained changes which had not yet been applied to the physical file. 

The `fixtool` command in the 14.2.1 release now has a -readonly option. When this option is used with an RFS protected file, `fixtool` will scan the file group by group using the system buffer checking for any errors. This ensures the most up-to-date file contents are being scanned and eliminates the requirement for the exclusive lock. For non RFS protected files, the physical file is scanned as with previous releases. 

**Note:** An error will be generated if the -readonly and -fix options are used together. Exclusive access to the file is still required when using the -fix option

## **try/catch exception handling** 

`try/catch` exception handling is a new mechanism for handling fatal errors using a new grammar similar to modern object-oriented languages. 

A `try/catch` statement defines a logical block. When a fatal error occurs in a BASIC statement or expression contained within the `try/catch` block, the `catch` statement will catch the error and handling of the exception will occur within the catch block scope. 

The `try/catch` statement is itself a BASIC statement and can be embedded into another `try/catch` block. This means that it supports cascading, or multi-level `try/catch` operations. 

## **Syntax** 

```
try
    <list of basic statements>
catch [<exception-variable>]
    <exception handler>
endtry
```

## **goto statement** 

- `goto` inside a `try/catch` block 

When a `goto` statement is outside the `try/catch` block and jumps to a target line that resides in the `try/catch` block (meaning it skips the running of the `try` statement), when an exception is thrown in the `try/catch` block, the `catch` statement will not catch the exception. - `goto` outside from a `try/catch` block 

When a `goto` statement that resides in a `try/catch` block jumps to a code line that is outside the `try/catch` block, the `try/catch` functionality will be terminated.

## **Dynamic Object programming** 

The Dynamic Object feature for UniVerse is the key feature that supports Object-Oriented Programming (OO Programming) patterns in the MV BASIC language. It supports most of the Object-Oriented concepts (similar to the JSON dynamic object system). There are several stages used to fully support Object-Oriented programming in Universe BASIC language. The dynamic object system is the stage that introduces the fundamental infrastructure for supporting OO Programming. 

The concept and features of dynamic objects in UniVerse are like the JavaScript language. It natively supports the JSON format. The dynamic object can easily be outputted as JSON format text and the dynamic object can be created from the JSON format text with built-in methods. There are rich built-in methods that help operate on dynamic objects, making it easy to use the dynamic object functionality. 

- Includes a brand-new grammar for creating and operating the Dynamic Object. 

- Dynamically extends the properties of the dynamic object. 

- Supports iterator operations against the dynamic object. 

- Supports a completely new `for` statement to iterate all properties of a dynamic object. 

- Supports output of dynamic object variables as JSON format text with `print` statement directly. 

- The value of a property can store BASIC variables of any type, not just atomic types the JSON format supports. 

- Supports a variety of built-in methods to operate on a dynamic object. 

- Supports built-in method with completely new grammar for normal BASIC variables. 

- Supports dynamic property accessing grammars.

## **Dynamic object type** 

There are two types of Dynamic Objects: 

- **Object type:** The concept of `object` type corresponds to the concept of the JSON format `object` type which can store different types of elements in its properties. The order of the properties is not related to the value of the object. This means that the two dynamic `object` objects have the same value even when they have different order of properties. Although there is no order required for the properties of a dynamic object, we can still define the order of properties when outputting the dynamic object as JSON format. The order of properties corresponding to the creation time of the property, the newer created property is always placed behind during output. There is also a built-in method to sort the properties of one dynamic object that changes the property order during output, but it does not change the value of the `object` . 

- Array type: The concept of the `array` type corresponds to the concept of the JSON format `array` type which can store different types of elements in the array. Unlike the `object` type, the elements in the array are ordered. When the order of an `array` type of dynamic object is changed, the value of the dynamic object array is changed.

## **Partial Write Recovery** 

Partial write recovery protects against data corruption when a user session performs a record level operation involving multiple data file block updates. If the session is terminated before the record level operation fully completes, the uvcleanupd process reverts all the updated data file blocks to their original contents. 

The partial write recovery process also handles situations where index file updates are involved. This can happen when a process terminates after the data file has been completely updated but before the indexes have been updated. In this situation, the recovery process will update the index files, maintaining the integrity of the data file with the index files. 

## **Note:** 

Partial write recovery is only available with RFS ON. By default, the partial write recovery uvconfig parameter MTX_ENABLED is set to 0 (OFF). 

To enable partial write recovery, you must enable RFS by setting RFS_MODE=1 and MTX_ENABLED=1 in the uvconfig file. 

When a partial write recovery event occurs, the result of the event is logged in the `uvcleanupd.log` file.

## **File Level Command Recovery** 

The File level command recovery performs a cleanup operation to ensure data files are left in a consistent state in the event a file level command terminates before completion. 

Previously, when a file level command terminated abnormally, it could result in the data file being left in an inconsistent state. Examples of such inconsistencies include a physical file existing with no corresponding VOC entry, or a temporary file being left behind. 

When a file level command terminates with file level command recovery, the uvcleanupd daemon will perform the cleanup work ensuring the file is left in a consistent state. 

As an example, a `DELETE.FILE` command is terminated prior to completion. Depending on the stage of execution, the cleanup work will either complete the delete operation or roll back the command to leave the file in the same state prior to the deletion. 

When a file level command recovery event occurs, the result of the event is logged to the `uvcleanupd.log` file.
