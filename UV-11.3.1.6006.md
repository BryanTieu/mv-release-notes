## Rocket UniVerse Hotfix 11.3.1.6006 

March 2017 

## Applicable platforms 

- All platforms 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at support@rocketsoftware.com. 

## Description of the problem 

|**Issue**<br>**number**|**Description**|
|---|---|
|UNV-5427|Windows only. Prior to this release, using an XML map with StartNode data containing a<br>trailing slash would cause the UniVerse session to crash. This issue has been fixed.|
|UNV-20769|Prior to this release, a sub-query in a SQL statement did not always use an index when<br>available. The selection process would scan the entire file rather than use the index. At<br>the current release, if an index is available it will be used during the selection process.<br>This issue has been fixed.|
|UNV-22254|Windows only. Previously, the`LISTU`command on Windows would incorrectly show<br>the uvcleanupd process as a phantom user. At the current release, it is correctly<br>displayed as the uvcleanupd process.|
|UNV-24384|Beginning at 11.1.0, running an`XMLExecute()`command in Basic with a very long<br>TCL command caused a uvsh core dump. In the reported example, the TCL command<br>was approximately 700,000 characters. This issue has been resolved.|
|UNV-24587|At the current release, U2 Replication has been updated to ensure all critical areas of<br>the code are properly protected from signal interruption.|
|UNV-24812|Windows only. Beginning at 11.3.1, in some circumstances uvcleanupd failed to<br>completely cleanup a UniVerse session. When a session was forcibly terminated,<br>uvcleanupd would release the locks and UniVerse license associated with the process.<br>However, the pid of that session would remain in the LCT table and be reported by<br>the`LISTUSER`and`PORT.STATUS`commands. A stop and restart of UniVerse was<br>required to clean the LCT table. This issue has been resolved.|
|UNV-24816|Beginning at 11.2.0, using the combination of the`XMAPCreate()`and`XMAPClose()`<br>functions in BASIC would result in a memory leak. The memory consumed during the<br>`XMAPCreate()`function call was not released when the handle was closed. This issue<br>has been fixed at the current release.|

Rocket UniVerse Hotfix 11.3.1.6006 

|**Issue**<br>**number**|**Description**|
|---|---|
|UNV-24899|At UnIVerse 11.3.1, logic was added to the UniVerse shutdown code on UNIX/Linux to<br>properly trap exit codes. This change resulted in customer-defined code added to the<br>end of the startup/shutdown script failing to run. This issue has been fixed and user-<br>defined code will now be executed.<br>Additionally, at this release a new feature has been added for running user-defined<br>code during UniVerse startup and shutdown. UniVerse will now look for a script named<br>`custom_uvrc.sh`in the`uvhome`directory. If found, the script will run.<br>UniVerse will add_start_and/or_stop_as the first argument when the script is called. For<br>example, on a shutdown the script will be called as`/usr/uv/custom_uvrc.sh`<br>`stop`. This will allow the script to be coded based on whether stopping or starting<br>UniVerse.<br>If custom-added code exists at the end of the script and the custom_uvrc.sh file is<br>found, only the custom_uvrc.sh file will run. The code at the end of the script will be<br>skipped. This is a deliberate design intended to avoid double execution of custom code.|
|UNV-24911|Starting at UniVerse 11.3.1, logic was added to the UniVerse shutdown script and the<br>installation script so that if UniVerse did not shutdown properly an upgrade would not<br>continue. However, in some rare situations, the installation would detect that UniVerse<br>was not shutdown completely and not allow the installation to continue. To avoid this<br>failure on an upgrade, the_-ssc_option is now available in the uv.load script and allows<br>users to skip the shutdown check.<br>If the installation process detects UniVerse is not shutdown completely, users will see<br>the following message:<br>`UniVerse did not shutdown properly.`<br>`Please bring UniVerse down completely.`<br>If you encounter this message, after confirming that UniVerse is down completely, re-<br>run the uv.load process with the_-ssc_option enabled. This will allow the installation to<br>proceed.|
|UNV-24919|Beginning at UniVerse 11.2.3, when performing a failover operation from publisher<br>to subscriber, U2 Replication may have failed to apply all pending updates to the<br>subscriber. If updates had reached the subscriber (Sub Got) but had not yet been<br>loaded into the replication buffer (Sub Available), these updates were lost during the<br>failover operation. The number of updates that were not applied and were lost is the<br>delta between the ‘Sub Available’ and ‘Sub Got’ LSN counts and this applied to each<br>replication group. This situation may have occurred in situations where the subscriber<br>system was overloaded and not keeping up with the publisher. Using the new Delayed<br>Standby replication feature could have exacerbated this situation and made it more<br>likely to occur.<br>Note: The LSN counts are viewable in the XAdmin Replication Status screen or via the<br>uvreptool command-line tool.<br>This issue has been resolved.|
|UNV-24962|Prior to this release, unirpc memory corruption may have occurred when using U2<br>Replication data link compression. When the size of the compressed data was larger<br>than the uncompressed data, memory corruption would occur, causing U2 replication<br>to crash. This could happen when the data contained information such as images. This<br>issue has been fixed.|

## Rocket UniVerse Hotfix 11.3.1.6004 

## December 2016 

## Applicable platforms 

- Windows 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at support@rocketsoftware.com. 

## Description of the problem 

|**Issue**<br>**number**|**Description**|
|---|---|
|UNV-24604|When a process was logging out, if a SIGUSR2 signal was received then a core dump<br>may have occurred. This issue is now resolved.|
|UNV-24706|Windows only. Prior to this release, the LOGOFF command would accept a pid that<br>corresponded to a UniVerse daemon process. If the pid entered was for the uvsmm<br>process, no new users would be able to log into the system once uvsmm is stopped.<br>Existing users might continue to run but a stop and restart of UV would be needed to<br>recover. At the current release, the LOGOFF command will no longer log out a pid that<br>corresponds to a UniVerse daemon process.|
|UNV-24720|Starting at 11.3.1, when using the 'AR' option in a LOCATE() function call and the value<br>used included periods and other non-numeric characters, the process would hang. This<br>issue is now fixed. If this situation is encountered, a string search will be done instead.|
