Review this information and complete the steps before upgrading to version 14.1.1. 

- UniVerse v14.1.1 will be installed with the System Buffer disabled and RFS Mode Off by default. 

- RFS two process mode has been removed in v14.1.1. 

- On AIX and Linux, the existence of the uvdb user is required for installation. In V14.1, the UniVerse daemon processes will run as the uvdb user. 

- On AIX and Linux, ACL support is required on the file system where UniVerse will be installed. This is necessary to support the daemon processes running as uvdb. 

- Python version has been upgraded to 3.11. 

- SSL version has been upgraded to OpenSSL 3.0.

## **Replication (HA/DR) Enhancements** 

The following replication (HA/DR) enhancements are included in UniVerse 14.1.1. 

## **New uv_repadmin skip option and new uvreptool option** 

A new `uv_repadmin` option has been added to allow for skipping an incomplete transaction that has caused replication to stop processing. This provides the administrator the option to keep Replication processing and avoids log build-up and the eventual reset and refresh due to the stalled group. The syntax is: 

```
uv_repadmin skip {-verbose} GROUP <group_name>
```

In addition to this new option, a new uvreptool option has been provided that allows changing the current value of the flags word for a replication LSN when choosing a group. This new option is primarily intended for use by Rocket Support when working with a customer who has reported that Replication has stopped processing. 

## **RW_QUEUESZ parameter in the repconfig file** 

The `repconfigfile` has been updated to accommodate a new parameter, RW_QUEUESZ. 

RW_QUEUESZ defines the size of the intelligent write queue. The intelligent write queue is used to optimize update locks and record updates on the subscriber when repeated updates of the same record are detected. 

RW_QUEUESZ can be set to a value between 0 and 65535.The default value is 0. This disables the intelligent write queue. If RW_QUEUESZ is set to a value larger than 65535, the value automatically defaults to 65535. 

## **Field Level Replication** 

Field-level replication saves system space by generating smaller replication files and reducing the amount of data being replicated between the publisher and subscriber. With field-level replication, only updated fields are written to replication logs instead of entire records. Various parameters and configurables, described in the sections that follow, affect field-level replication. 

Field-level replication occurs under the following conditions: 

- `FIELDWRITE` statement in the application generates field-level updates. The `FIELDWRITE` statement allows you to update the specified fields in an existing record. For those using replication, `FIELDWRITE` generates field-level replication logs. 

- `WRITEV` statement generates field-level updates. The `WRITEV` statement updates a specified attribute or attributes in a file regardless of lock status. The `WRITEV` statement releases locks set by the same process. 

- The FIELD_UPDATE_THRESHOLD parameter is configured so that WRITE commands generate field-level updates. 

- You define field-level replication objects using the FILE phrase. For more information, see the FILE documentation in Replication configuration file. 

For more information on Field Level Replication, see the _UniVerse 14.1.1 Replication User Guide_ and the _UniVerse 14.1.1 BASIC Commends Reference Guide_ .

## **Subscriber accepting logs from an older protocol level** 

**Note:** Although this feature was introduced in UniVerse v11.3.5, we are also providing this information in the 14.1 release to give this new feature greater visibility. 

Previously,when using U2 Replication, the publisher and subscriber were required to be running the same Replication protocol level for Replication to function correctly. Starting in 11.3.4.9006, U2 Replication has been enhanced to allow a subscriber to accept replication logs from a publisher system running 11.3.1. This will allow a customer to upgrade the subscriber system to 11.3.4.9006 or 11.3.5 and have the subscriber system still accept replication logs from an 11.3.1 publisher. When running in this configuration, some things to note: 

- A failover operation to a hot standby subscriber running with this new functionality release will be successful. 

- Replication between the failed over publisher will be suspended with any subscribers running the 11.3.1 release. 

- A failback to the original publisher will not be possible if the publisher is still running the 11.3.1 release. 

- The 11.3.4.9006 or 11.3.5 system running with the newer protocol level can accept logs produced on a system running with the 11.3.1 protocol level. However, the reverse is not possible. An 11.3.1 system cannot accept logs from a publisher system running 11.3.4.9006 or 11.3.5. 

See the _Replication protocol level_ topic in the _UniVerse Data Replication User Guide_ for more information.

## **RFS Off mode** 

In v14.1.1 the Recoverable File System (RFS) mode is set to OFF (0) by default. 

**Note:** RFS ON will be available in the v14.1.2 release. 

- The RFS two process architecture has been removed in v14.1.1. 

- The SYSTEM_BUFFER uvconfig parameter has been removed in v14.1.1. 

- UniVerse behavior in 14.1 with RFS Mode OFF: 

   - When a UniVerse file is opened, an AFT entry is created with some key information such as inode and dno numbers. This information is used for concurrency control. 

   - File header information will not be stored in the AFT entry. 

   - File activity will not be cached to the System Buffer. File I/O will be applied directly to the physical file as was done in UniVerse releases prior to 12.1. 

   - All UniVerse files are configured as unprotected. This cannot be changed with RFS OFF. 

Starting with UniVerse v14.1.1, the system buffer runtime recovery framework has been introduced to ensure consistency and integrity in critical regions of the database for improved stability of the UniVerse server. Some important things to note about the system buffer runtime recovery framework: 

- Performs recovery for shared data structures keeping them consistent when a UniVerse session terminates abnormally or when a child process is killed. 

- The recovery work is done by Uvcleanupd. 

- No configuration is required for the recovery framework.

## **OpenSSL version 3.0 upgrade** 

The version of OpenSSL shipped with UniVerse in this release has been upgraded to 3.0. 

There are several important changes to be aware of: 

- The SSLv3 protocol is not available in OpenSSL 3.0 and has been removed from UniVerse, including the option for the uvconfig parameter: SSL_PROTOCOLS. 

- The provided U2SelfCert.scr security context record has been updated to use the TLSv1.2 protocol at a minimum. Connections using this SCR will need to support the TLSv1.2 protocol. 

- A new SSL_SECURITY_LEVEL parameter has been added to uvconfig. The default OpenSSL security level is 1. This has the following affects: 

   - Protocols TLSv1 and TLSv1.1 are disabled at security level 1. Change the SSL_SECURITY_LEVEL parameter in uvconfig to 0 to use these protocols. 

   - X509 certificates signed using SHA1 or MD5 are disabled at security level 1. Rocket suggests re- signing certificates with safer algorithms such as sha256, sha512. 

      - Setting SSL_SECURITY_LEVEL to 0 in uvconfig will allow the use of certificates signed with SHA1 or MD5. For example, attempting to use the addCertificate() function on a certificate signed by either SHA1 or MD5, with the usedAs parameter set to SSL_CERT_SELF, this function will fail. 

- Some encryption algorithms are now considered unsafe and are treated as 'legacy' algorithms in OpenSSL 3.0. These include: 

   - Hashing Algorithms / Message Digests: MD2, MD4, MDC2, WHIRLPOOL, RIPEMD160 

   - Symmetric Ciphers: Blowfish, CAST, DES (DES_ECB, DES_CBC, DES_OFB, DES_CFB, DES_CFB1, DES_CFB8 and DESX_CBC), IDEA, RC2, RC4, SEED 

   - Key Derivation Function (KDF): PBKDF1 

- A new parameter has been added to uvconfig: USE_LEGACY_OPENSSL_ALGORITHM. A setting of 1 will allow the use of these algorithms. 

- A new TCL command has been added: `USE.LEGACY.OPENSSL.ALGRORITHM {ON|OFF| INFORM} {BRIEF|HUSH}` . This command will allow the use of these algorithms in the current session. 

- Secure Renegotiation (RFC5746) is enforced in UniVerse 14.1.1. To control whether or not UniVerse as a client can connect to a server that does not support secure renegotiation, options have been added to the SSL_OPTIONS parameter in uvconfig: NO_LEGACY_SERVER_CONNECT (default) and LEGACY_SERVER_CONNECT. 

- If customers use the `SIGNATURE()` function to verify data that is signed by the MD4 algorithm using OpenSSL 1.1.1 or 1.0.2, this function will fail due to OpenSSL incompatibility. 

- SSL_VERSION_PATH can however be used to call an OpenSSL version other than the version provided with UniVerse (only OpenSSL 3 can be used). However, SSL_VERSION_PATH cannot be used to enable FIPS mode in OpenSSL 3.0 by pointing to the OpenSSL 3.0 on RHEL9. 

For more information, see the _UniVerse 14.1.1 Security Features Guide_ .

This release notes document contains a cumulative summary of all the Hotfixes made available for a specific Generally Available (GA) product version. Please browse this document to find the Hotfix release that fits your specific needs. 

Hotfixes are identified as "Controlled Releases" within Rocket Business Connect (RBC). Hotfixes are only visible by searching for the specific version and build number in the Controlled Release field. 

To obtain a specific Hotfix: 

1. Log into the RBC system and modify your existing license. 

2. Click **Edit** next to your existing product, and then click **Search** to view the list of GA product releases. 

3. After the product availability matrix displays, enter the Hotfix version number (which includes the product version and build number, for example 11.3.1.6005) in the **Controlled Release** field just above the table of product releases. 

4. Click **Search** . 

The Hotfix release will display on the product matrix, and can be selected for ordering and download. If you have questions about RBC, please contact us at U2Support@rocketsoftware.com. 

**Note:** Hotfix releases are intended to be short-term solutions and have had limited testing in order to be made available quickly. All Hotfix changes will be included in the next GA release, and will be fully tested by Rocket’s Quality Assurance (QA) team across all supported platforms and environments. It is strongly recommended that after taking a Hotfix, customers upgrade to the next available GA release to ensure the highest quality experience.
