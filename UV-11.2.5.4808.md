## Rocket UniVerse Hotfix 11.2.5.4808 

## Version 11.2.5.4808 

## Applicable platforms 

- AIX 

- HP-UX Itanium 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at U2Support@rocketsoftware.com. 

## Description of the problem 

|**Issue number**|**Description**|
|---|---|
|UNV-22653|Prior to this release, after using the break key, attempts to start a child process<br>with the`SH`command would hang if a`COMO`file was active. The behavior was not<br>apparent on all platforms. This issue has been resolved.|
|UNV-22862|Prior to this release, orphaned UniVerse processes waiting on a mutex lock might<br>be left behind under certain circumstances. The behavior involved running a<br>BASIC program calling GCI subroutines and using the break key. This problem has<br>been resolved.|
|UNV-23221|HP Itanium only. Prior to this release, the unlock command may have failed to<br>release a stuck group lock or login lock semaphore. When this occurred, stopping<br>and restarting UniVerse was the only method for releasing the semaphore. This<br>issue has been resolved.|
|UNV-23288|Prior to this release, the`CREATE.INDEX`command may have created the index<br>directory with incorrect permissions. If the user creating an index for the first time<br>on a data file was not the owner of the file, the index directory was created with<br>full read/write permissions. At the current release, the 'rw' permissions on the<br>index directory are correctly based on the permissions of the data file.|
|UNV-23454|Prior to this release, the`CREATE.INDEX`command added execute permissions<br>to user/group/other on the`I_filename`directory, regardless of data file<br>permissions. Now, the execute permissions are added to the`I_filename`<br>directory based on the 'r' permission settings on the data file.|

## Rocket UniVerse Hotfix 11.2.5.4805 

## Version 11.2.5.4805 

## Applicable platforms 

- 64-bit Linux 

- 32-bit Linux 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at U2Support@rocketsoftware.com. 

## Description of the problem 

|**Issue number**|**Description**|
|---|---|
|UNV-11596|The SYSTEM(105) BASIC function reports on the NLS configuration parameters in<br>uvconfig. Prior to this release, the value for NLSDEFSOCKMAP was not reported.<br>At this release, the value of NLSDEFSOCKMAP is reported in attribute 18 of the<br>result. The EQUate value for this parameter is NLS$DEFSOCKMAP.<br>**Note:**To use the EQU values, the UVNLS.H file must exist in the INCLUDE file.|
|UNV-22113|Previously, installing NLS through the UNIX/Linux UV System Admin menu<br>(Package->Install Package) would cause the report to show that a record was<br>locked by another user. This locked record caused the NLS Administration menu<br>to not be added to the Package menu. The code for the installation process has<br>been changed so that the record lock is not needed. Starting at this release, the<br>NLS administration menu will appear as expected in the System Admin Package<br>menu.|
|UNV-23037|Prior to this release, a memory leak existed when accessing an index file that was<br>created with the NLS COLLATE option. The memory leak was apparent on any<br>type of access and did not require an index update. This issue has been resolved.|
|UNV-23072|On UNIX/Linux, National Language Support (NLS) can now be installed during<br>the installation/upgrade process. To enable NLS installation, specify the`-nls`<br>command-line option. On installation, the NLS package will be installed with<br>NLSMODE set to 1 and enabled with uvregen.<br>On upgrades where NLS was previously installed, the upgrade will abort.<br>Traditionally, NLS should be uninstalled before performing an upgrade. To<br>override this check on upgrades, use the`-fnls`(force nls) option instead. The<br>uvconfig NLSMODE parameter is not changed on upgrades.|
|UNV-23081|Linux only. Prior to this release, UniVerse might terminate unexpectedly when<br>using NLS. If invalid NLS data was present, using the BASIC TRIM function with<br>this data could cause UniVerse to abnormally terminate. This issue has been<br>resolved.|

## Rocket UniVerse Hotfix 11.2.5.4803 

## Version 11.2.5.4803 

## Applicable platforms 

- 64-bit Sun Solaris Intel 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at U2Support@rocketsoftware.com. 

## Description of the problem 

|**Issue number**|**Description**|
|---|---|
|UNV–21960|Due to a problem in how the byte order of the Solaris Intel platform was<br>initially defined, various problems could be encountered when moving data<br>from another UniVerse platform to Solaris Intel. The issues primarily involved<br>situations involving basic object code. This includes dictionaries based on I-<br>types, distributed file partitioning algorithms, and BASIC subroutines. Abnormal<br>terminations could be encountered on the Solaris Intel platform when accessing<br>virtual field indexes, accessing distributed files, using uvrestore to restore<br>distributed files, and accessing UVNet files with an I-type dictionary.<br>The definition of the byte order on the Solaris Intel platform has been resolved at<br>the current release.<br>The fix for this issue also resolves the following issues: UNV-20417, UNV-21315,<br>UNV-21346, and UNV-21348.|

## Rocket UniVerse Hotfix 11.2.5.4802 

## Version 11.2.5.4802 

## Applicable platforms

- HP Itanium 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at U2Support@rocketsoftware.com. 

## Description of the problem 

|**Issue number**|**Description**|
|---|---|
|UNV-22851|Beginning at this release, the UniVerse XML parser will only allow well-formed<br>XML documents. Applications that do not meet current XML standards may not<br>be able to use the XDOMTransform functionality. General XML functionality<br>may also be impacted if documents do not follow current XML standards.<br>Current XML documents encountering problems must be updated before using<br>XDOMTransform functionality.|
|UNV-22394|Prior to 11.2.0, the UniVerse installation included the Xalan open source XML-<br>related library. At 11.2, the Xalan library was replaced with the XQilla library due<br>to support for Xalan no longer being available. This resulted in problems for<br>UniVerse applications using certain XSLT functionality such as 'include' which is<br>not yet available in the XQilla library.<br>Beginning at this release, the UniVerse installation contains both the Xalan and<br>XQilla libraries, restoring the missing XLST functionality. This issue is resolved.|
