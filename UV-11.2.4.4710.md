## Rocket UniVerse Hotfix 

Version 11.2.4.4710 

Applicable platforms 

- 64–bit SOLARISINT 

- 64–bit SOLARIS 

- 64–bit LINUXX86 

- 64–bit HPUXITANIUM 

- 64–bit AIX 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at U2Support@rocketsoftware.com. 

Description of the problem 

|**Issue number**|**Description**|
|---|---|
|UNV-13746|Beginning at this release, the ODBCLOGIN subroutine has been added to<br>UniVerse. The ODBCLOGIN subroutine allows users to create a global subroutine<br>that can be used to control access to ODBC connections based on the user<br>ID making the connection. See Tech Note UNV-20488 for more details on the<br>ODBCLOGIN subroutine.|
|UNV-15416|Prior to this release, some character sequences failed to display properly when<br>COMMAND.EDITOR was being used. For example, the entry of CHAR(241) would<br>be echoed as CHAR(7) when COMMAND.EDITOR was turned on. This issue has<br>been resolved.|



Rocket UniVerse Hotfix 

|**Issue number**|**Description**|
|---|---|
|UNV-19579|An enhancement has been made to the Audit log. Beginning at this release, it is<br>possible to include the full path of the program being executed in the Audit log<br>file. You can also optionally include the the stack of programs to the Audit log<br>file.<br>There are two global configuration policies:<br>`log_program_path= ON | OFF default to ON`<br>`log_program_stack=ON | OFF default to ON`<br>These two policies are independent of one another.<br>If log_program_path is ON, all program/subroutine names will be full path.<br>For example,<br>`Program........... /eskimo3/uv112/catdir/*GSUB2.19579/FSUB2`<br>If log_program_stack is ON, the full execution stack will be logged with the<br>current program path. The PROGRAM field will be a multi-value field. The first<br>value is the current program, the second value is "Call Stack:", the rest of values<br>are stack contents in reverse calling order, starting from the most recent called<br>program/subroutines, to the main program. For example,<br>`Program.................. /eskimo3/uv112/catdir/*IS.FULLPATH`<br>`. Call Stack:`<br>`. /eskimo3/uv112/catdir/*GSUB2.19579/FSUB2`<br>`. /eskimo3/uv112/catdir/*GSUB2.19579/FSUB1`<br>`. /eskimo3/uv112/catdir/*GSUB2.19579/`<br>`. /home/jcui/uvtest/BP.O/19579/SUB2`<br>`. /home/jcui/uvtest/BP.O/19579/SUB1`<br>`. /home/jcui/uvtest/BP.O/19579/`<br>NOTE: If you have nested or recursive programs, this log can grow very long and<br>overall performance may be negatively impacted.|
|UNV-19580|Beginning at this release, when using Audit logging to log database updates,<br>the audit logs have been enhanced to distinguish between a new record<br>being added (insert) to the database and an existing record being modified<br>(update). For example, the audit log entry will now contain either WRITE:ADD or<br>WRITE:UPDATE to distinguish the type of write operation performed.|
|UNV-19630 and<br>UNV-19631|An enhancement has been made to the DAT.QUERY.COMMAND and<br>DAT.QUERY.RESULTSET events. Now, when the DAT.QUERY.RESULTSET event<br>is enabled, the DAT.QUERY.COMMAND audit log record contains the command<br>executed and the result set containing all record IDs that were returned by the<br>command. Note, the individual DAT.QUERY.RESULTSET audit log entries that<br>contained the actual data returned by the command are no longer written to the<br>audit log.|
|UNV-20288|For Linux and HP Itanium only at UniVerse 11.2.4, if the LANG environment<br>variable was set to a value other than C, the unirpcservices file might be modified<br>incorrectly on an 11.2.4 upgrade. Starting at this release, the LANG setting is<br>temporarily changed to C during the installation process, which allows the<br>unirpcservices file to be updated properly, fixing this issue. The workaround at<br>UniVerse 11.2.4 is to manually set LANG=C prior to running uv.load.|
