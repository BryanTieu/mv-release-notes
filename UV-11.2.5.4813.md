## Rocket UniVerse Hotfix 11.2.5.4813 

## Version 11.2.5.4813 

March 2016 

## Applicable platforms 

▪ AIX 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at U2Support@rocketsoftware.com. 

## Description of the problem 

|**Issue number**|**Description**|
|---|---|
|UNV-23668|Beginning at 11.2.3.4641, in the U2 CallMQI API, when a message ID or correlation<br>ID had a null character in its twenty-four bytes, CallMQI would truncate the ID at<br>the null character when presenting it to a BASIC program. This issue has been<br>resolved.<br>**Note:**This issue was introduced in code changes related to release 11.2.3.4641.<br>Versions of UniVerse between 11.2.3.4641 and this release could experience<br>interoperability issues if exchanging IBM MQ messages between each other via<br>CallMQI.<br>In particular, this could affect systems using the Request/Response messaging<br>style. An issue could arise if the current UniVerse release generates a Request<br>message with a message ID that contains null characters, or is less than twenty-<br>four bytes in length (the size of an IBM MQ message ID), and then one of the<br>previously listed versions of UniVerse processes that Request message and<br>generates a Response message back, where the Response message's correlation<br>ID is set to the Request message’s message ID. In this scenario, the Response<br>message’s correlation ID will be a truncated version of the original message ID. If<br>the current UniVerse system then waits on the Response message, looking for a<br>correlation ID that matches its original message ID, it will fail to find the Response<br>message.|



|**Issue number**|**Description**|
|---|---|
|UNV-23716|Prior to this release, customers may have experienced situations where<br>connection pool (CONNPL) licenses were being consumed by process IDs that<br>were not associated with existing uvapi_slave processes. When this happened,<br>the only way to release the licenses back to the available pool of licenses was to<br>stop and restart UniVerse.<br>Several changes have been made at the current release related to this situation.<br>▪<br>Changes have been made to the uvapi_slave process such that when the<br>process exits, the connection pool license held by the process will always be<br>released.<br>▪<br>The "clean_lic" option of uvlictool has been re-enabled at this release. When<br>this option is used, uvlictool will scan the license table and release any license<br>associated with a pid, which is not a valid UniVerse process.<br>▪<br>The uvcleanupd process has been enhanced to perform the same function<br>as the "clean_lic" option of uvlictool when it runs on its regularly scheduled<br>interval.<br>This issue has been resolved.|
