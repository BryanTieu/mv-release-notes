## General Hotfix information 

This release note contains a cumulative summary of all the Hotfixes made available for a specific Generally Available (GA) product version. Please browse this document to find the Hotfix release that fits your specific needs. 

Hotfixes are identified as "Controlled Releases" within Rocket Business Connect (RBC). Hotfixes are only visible by searching for the specific version and build number in the Controlled Release field. 

To obtain a specific Hotfix: 

1. Log into the RBC system and modify your existing license. 

2. Click **Edit** next to your existing product and then click **Search** to view the list of GA product releases. 

3. After the product availability matrix displays, enter in the Hotfix version number (which includes the product version and build number, for example 11.3.1.6005) in the **Controlled Release** field just above the table of product releases, then click **Search** . 

The Hotfix release will display on the product matrix and can be selected for ordering and download. If you have questions about RBC, please contact us at U2Support@rocketsoftware.com. 

**Note:** Hotfix releases are intended to be short-term solutions and have had limited testing in order to be made available quickly. All Hotfix changes will be included in the next GA release, and will be fully tested by Rocket’s Quality Assurance (QA) team across all supported platforms and environments. It is strongly recommended that after taking a Hotfix customers upgrade to the next available GA release to ensure the highest quality experience. 

For questions, please contact support at U2Support@rocketsoftware.com .

## Rocket UniVerse Hotfix 12.1.1.1012 

## December 2019 

## Applicable platforms 

- Windows 

- Linux 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at support@rocketsoftware.com. 

## Description of the problem 

|**Issue**<br>**number**|**Description**|
|---|---|
|UNV-28540|Prior to this release, UDO BASIC functions were not handling large integer values<br>correctly in all cases. This issue has been resolved in the current release. Integer input<br>data in the range -2^64 to 2^64 will be treated as 64bit integers.|
|UNV-28642|Previously, a dynamic file split/merge operation may have left a record lock set on the<br>incorrect group. The problem was a timing issue related to updating the dynamic file<br>information in the T30FILE table. The issue has been resolved with this release.|
|UNV-28654|Beginning with 12.1.1, the`audman`command could no longer be run by the uvadm<br>user. A message was displayed indicating the user did not have the proper privileges to<br>run the command. This issue has been resolved.|
|UNV-28690|Windows only. Beginning with 12..1.1, using the BASIC`LOCATE`statement with the`AR`<br>sort option might hang under certain conditions. If the data being searched contained a<br>decimal point and non-numeric data (for example, "12345.ABC"), the statement would<br>hang. This issue has been resolved.|
|UNV-28872|Windows only. Beginning with 12.1.1, the use of CTRL+C to interrupt a UniVerse session<br>did not always interrupt the session. This issue has been resolved.|
|UNV-28910|Beginning with 12.1.1, executing the uv.rc script in`/etc`without an argument would<br>incorrectly fail with an invalid option message. In previous releases, executing uv.rc<br>without any options would default to starting UniVerse. The previous behavior has been<br>restored, resolving this issue.|

## Rocket UniVerse Hotfix 12.1.1.1011 

## September 2019 

## Applicable platforms 

- Windows 

- Linux 

- AIX 

This download contains the most recent Hotfix for the previously released product. For questions, please contact support at support@rocketsoftware.com. 

## Description of the problem 

|**Issue**<br>**number**|**Description**|
|---|---|
|UNV-25032|Prior to this release, the`BUILD.INDEX`command leaked memory when the<br>CONCURRENT keyword was used. Depending on the size of the index, the<br>`BUILD.INDEX`command could exit unexpectedly due to running out of memory. This<br>issue has been resolved.|
|UNV-28324|Windows only. Prior to this release, if the UniRPC Service was started as a user other<br>than SYSTEM, the output of LISTU reports UniRPC connections with ownership as the<br>user starting the service.<br>At this release, the LISTU results will show the correct username.<br>**Note:**If SYSTEM is not used as the starting process, the XAdmin Telnet Configuration<br>functionality will not work on UAC-enabled systems. This is because write permissions<br>are needed to access the UniVerse registry keys.|
|UNV-28579|Beginning at 12.1.1, record data could be lost when using the BASIC`WRITEV`or<br>`FIELDWRITE`statements on a file with a trigger.<br>The new.record argument passed to the trigger subroutine would contain only those<br>fields specified in the`WRITEV`or`FIELDWRITE`statement. Additionally, the record<br>written by these statements would only contain the specified fields. All other fields that<br>previously existed in the record would be missing. This issue has been resolved.|
|UNV-28589|Beginning at 12.1.1, the**sidcache.exe**binary used by the uvdiag tool was renamed to<br>**uvsidcache.exe**.<br>At this release, the**sidcache.exe**binary from prior releases will be removed during<br>an upgrade install. Additionally, the uvdiag tool has been updated to now use the<br>**uvsidcache.exe**binary when uvdiag is executed.|
|UNV-28600|In some cases, the memory allocated for RAID execution has been increased again to<br>better handle debugging when indexed complex I-Types are used.<br>This was previously changed via UNV-24653 at UniVerse 11.3.1.|

## Applicable platforms

## Description of the problem 

|**Issue**<br>**number**|**Description**|
|---|---|
|UNV-28152|Prior to this release, the colon (:) character in a BASIC CallHTTP request would be<br>encoded to '%3A'. This could cause failures when connecting to certain sites. In<br>the current release, the default behavior is to not encode the colon character. If<br>there is a requirement to encode the colon character, the environment variable<br>COLON_ENCODES can be set.|
|UNV-28185|Prior to this release, the behavior of the BASIC`GET`statement was inconsistent when<br>used with`TIMEOUT`on an NLS enabled system. The`GET`statement might timeout at<br>an interval different from that specified in the`TIMEOUT`statement. This issue has been<br>resolved.|
|UNV-28270|Windows only. Prior to this release, the BASIC`XDOMLocate`function could crash when<br>encoding was set to UTF-8 by the`XMLSETOPTIONS`command. This issue has been<br>resolved.|
|UNV-28313|Beginning with 12.1.1, the performance of the`BUILD.INDEX`command was<br>negatively impacted under certain circumstances. If the index being built resulted in<br>a very large number of keys being associated with an index value, the performance of<br>`BUILD.INDEX`was degraded from prior releases. This issue has been resolved.|
